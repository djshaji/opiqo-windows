# WASAPI Implementation Issues

All 8 issues have been fixed in `src/win32/AudioEngine.cpp`.

---

## Fixes Applied

### 1. `stereoBuf` Buffer Overflow — **Fixed**
`stereoBuf` was allocated as `renderBufSize * 2` floats, but `engine->process()` uses the same buffer for both input and output halves, requiring `renderBufSize * 4`.

**Change:** `stereoBuf.assign(renderBufSize * 4, 0.f)` (later updated to use `captureBufSize * 4` after fix 6).

---

### 2. Exclusive Mode Broken with `period=0` — **Fixed**
`IAudioClient::Initialize` in exclusive mode requires a non-zero `hnsBufferDuration`. Passing `0` returns `AUDCLNT_E_INVALID_SIZE`.

**Change:** Before initializing in exclusive mode, `GetDevicePeriod` is now called on each client to retrieve its minimum supported period. That value is passed as both `hnsBufferDuration` and `hnsPeriodicity`. Shared mode continues to pass `0`.

---

### 3. `sampleRate` Parameter Silently Ignored — **Fixed**
The negotiated device sample rate was never validated against the caller-requested rate.

**Change:** After `GetMixFormat`, the capture and render sample rates are compared against `impl_->sampleRate`. A mismatch transitions to `State::Error` with a descriptive message. `impl_->sampleRate` is then updated to the confirmed negotiated rate.

---

### 4. Shared Capture/Render Event Handle — **Fixed**
Both `captureClient` and `renderClient` were given the same event handle, causing render-side wakeups to spuriously fire the capture loop.

**Change:** A second `HANDLE renderEvent` field was added to `Impl`. Each client now receives its own dedicated auto-reset event. `renderEvent` is created, set on the render client, and closed in `releaseResources()`.

---

### 5. Incomplete PCM Format Support — **Fixed**
`toFloat`/`fromFloat` only handled IEEE float and 16-bit PCM. 24-bit and 32-bit integer formats produced garbage audio.

**Change:** Both conversion functions now dispatch on `wBitsPerSample`:
- **32-bit int** — scaled by ±2,147,483,648
- **24-bit packed PCM** — 3-byte little-endian, sign-extended, scaled by ±8,388,608
- **16-bit PCM** — existing fallback, scaled by ±32,768
- **Float** — direct `memcpy` (unchanged)

---

### 6. Capture Buffer Can Overflow `inBuf` — **Fixed**
`inBuf` and `outBuf` were sized using `renderBufSize`, but capture packets are delivered at the capture device's own period, which can exceed the render buffer size.

**Change:**
- `captureBufSize` is now obtained via `captureClient->GetBufferSize()`.
- `inBuf` is sized `captureBufSize * capChannels`.
- `outBuf` is sized `max(captureBufSize, renderBufSize) * renChannels`.
- `stereoBuf` is sized `captureBufSize * 4`.

---

### 7. Device IDs Passed via `errorMsg` Field — **Fixed**
Device IDs were encoded into `errorMsg` as a `|`-delimited string before the audio thread launched, creating a fragile implicit contract and a potential race window.

**Change:** Two dedicated fields `inDeviceId` and `outDeviceId` were added to `Impl`. `start()` writes them directly; `audioThreadProc()` reads them directly. The `errorMsg` carrier hack was removed.

---

### 8. `errorMsg` Has No Thread Safety — **Fixed**
`impl_->errorMsg` was written from the audio thread and read from the calling thread with no synchronization — a data race under the C++ memory model.

**Change:**
- `mutable std::mutex errorMsgMutex` added to `Impl`.
- `AudioEngine::errorMessage()` now locks the mutex before reading.
- A `setErr` lambda defined at the top of `audioThreadProc()` locks the mutex before every write. All 14 direct assignments in the audio thread were replaced with `setErr(...)`.

---

## Additional Fixes

### 9. No-Plugin Chain Produces Stale Output — **Fixed**
**Location:** `src/LiveEffectEngine.cpp` — `process()`

When all four plugin slots were empty (or all disabled), the plugin loop completed without writing to `output`, leaving it with whatever data remained from the previous callback — effectively stale audio or silence on the first call.

**Change:** A boolean `anyProcessed` flag tracks whether any plugin ran. If none did, `memcpy(output, input, ...)` is called at the end of the loop — consistent with the `bypass` and `try_lock` fast-paths above it.

---

### 10. Gain Slider Has No Effect on Audio — **Fixed**
**Location:** `src/LiveEffectEngine.cpp` — `process()`

`liveEngine_.gain` is a `float*` allocated in the constructor and updated by `onGainChanged()` in `MainWindow`, but `process()` never read it — the gain value was stored but never applied to any audio samples.

**Change:** After the plugin chain (and the no-plugin pass-through fallback), the output buffer is scaled by `*gain` when the value differs from `1.0f`. The `g != 1.0f` guard skips the multiply loop at unity gain.
