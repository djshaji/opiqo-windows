# Issue 32: Writable Atom Sequence File Path Passing — Bugs & Analysis

## Summary

Analysis of the pipeline used to send file paths to LV2 plugins via `patch:Set` atom messages
(`setFilePath` → `send_path_parameter` → `process()` Step B).

**Status: All issues resolved.**

---

## Bug 1: Windows path guard silently rejects all valid paths (critical) ✅ Fixed

**File:** `src/LV2Plugin.hpp` — `send_path_parameter`

The original guard `if (abs_path[0] != '/')` unconditionally rejected every Windows absolute
path (`C:\...`, `\\server\...`), silently blocking all file-passing to plugins on Windows.

**Change applied:**

```cpp
#if defined(_WIN32)
    const bool is_abs = (std::strlen(abs_path) >= 2 && abs_path[1] == ':') ||
                        (abs_path[0] == '\\' && abs_path[1] == '\\');
#else
    const bool is_abs = (abs_path[0] == '/');
#endif
    if (!is_abs) {
        LOGE("send_path_parameter: expected absolute filesystem path, got: %s", abs_path);
        return;
    }
```

---

## Bug 2: `setFilePath` reads atom response before any `process()` cycle runs ✅ Fixed

**File:** `src/LiveEffectEngine.cpp` — `setFilePath`

The `readAtomMessage("notify", msg)` block immediately after `send_path_parameter` always
returned false (the plugin cannot respond until `lilv_instance_run()` runs in the next audio
callback), logging a spurious error on every file load.

**Change applied:** Removed the `readAtomMessage` block entirely. Replaced with a `LOGD`
confirming the message was queued and will be delivered on the next `process()` cycle.

---

## Issue 3: Redundant atom port scan in `setFilePath` ✅ Fixed

**File:** `src/LiveEffectEngine.cpp` — `setFilePath`

The manual loop over `p->ports_` to find the atom input port (used only for a log line) was
redundant — `send_path_parameter` performs the same scan internally.

**Change applied:** Removed the loop and associated `port` variable and log line.

---

## Issue 4: Default `required_atom_size_` (8192) may be too small for long model paths ✅ Fixed

**File:** `src/LV2Plugin.hpp`

Long Windows paths could push the forged `LV2_Atom_Event` past the sequence buffer capacity,
causing the event to be silently dropped in `process()`.

**Changes applied:**

- Raised `required_atom_size_` default from `8192` to `65536` in both constructors, so all
  atom port buffers start at 64 KB regardless of whether the plugin declares `lv2:minimumSize`.
- Added an early-warning `LOGE` in `send_path_parameter` that fires immediately if the
  payload would overflow the port buffer, diagnosing the problem at the call site rather than
  silently inside `process()`.

---

## Notes

- The atom body packaging itself was correct and unchanged: `send_path_parameter` stores only
  the object body (`LV2_ATOM_BODY`) and records `atom->type` as the message type; `process()`
  correctly reconstructs a valid `LV2_Atom_Object` inside the sequence.
- The worker thread pipeline (`lv2:Worker`) is unrelated to this flow and was not affected.
