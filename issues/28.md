## Fix 1 — Sample rate hard-fail regression

**File:** `src/win32/AudioEngine.cpp`

The strict validation added in a previous session hard-failed if the capture device rate or render device rate didn't match the requested rate. On Windows, many systems have capture at 44100 Hz and render at 48000 Hz; this caused the engine to immediately enter `State::Error`.

**Fix:** Removed the hard-fail block entirely. WASAPI shared mode negotiates the mix format independently per endpoint — the OS owns the format and it cannot be overridden. `impl_->sampleRate` is now updated to `renderFmt->nSamplesPerSec` (authoritative for the DSP chain) without any validation error.

## Fix 2 — Wine dropped frames / render starvation

**File:** `src/win32/AudioEngine.cpp`

When the render buffer had less available space than the number of captured frames (common under Wine's less-precise WASAPI timing emulation), the code used `toWrite = std::min(numFrames, available)` and silently discarded the excess frames. This starved the render buffer in subsequent periods, causing a cascade of dropouts.

**Fix:** The render write path now:
- Skips the write entirely (counting a dropout) if `available == 0`
- Writes `min(dspFrames, available)` frames
- Pads the remainder with `AUDCLNT_BUFFERFLAGS_SILENT` silence frames so the render timeline stays intact and the next period starts with a full buffer

## Fix 3 — Speex resampler for capture/render rate mismatch

**Files:** `src/win32/AudioEngine.cpp`, `CMakeLists.txt`

When capture and render devices run at different sample rates (e.g. Wine: 44100 Hz capture, 48000 Hz render), audio was passed at the wrong rate, producing pitch-shifted or distorted output.

**Fix:** Added `resample.c` (speex resampler, standalone build) to the project and wired a `SpeexResamplerState*` into `AudioEngine::Impl`. If `captureFmt->nSamplesPerSec != renderFmt->nSamplesPerSec` at stream start, a quality-5 resampler is created and used in `runLoop()` to convert the stereo downmix from capture rate to render rate before DSP and render write. The resampler is destroyed in `releaseResources()`.

CMake changes:
- `src/resample.c` added to `ENGINE_SOURCES`
- `set_source_files_properties` applies `-DOUTSIDE_SPEEX -DRANDOM_PREFIX=opiqo` to avoid symbol clashes

New `Impl` fields:
- `SpeexResamplerState* captureResampler` — nullptr when rates match
- `std::vector<float> resampledBuf` — resampler output at render rate
- `std::vector<float> dspOutBuf` — DSP output (was `stereoBuf` second half)

## Fix 4 — "Auto" sample rate option in Settings

**Files:** `src/win32/SettingsDialog.cpp`, `src/win32/MainWindow.cpp`

The sample rate combo only offered fixed values (44100 / 48000 / 96000). When using mismatched devices, users had to know the correct rate; with the new resampler there is no need to specify one.

**Fix:** Added an "Auto" entry as index 0 in the sample rate combo. Selecting it stores `sampleRate = 0` in `AppSettings`. The audio engine already overwrites `impl_->sampleRate` from the driver-negotiated rate at stream start, so `0` acts as a clean "let the driver decide" sentinel.

The first-run auto-detect block in `MainWindow::create()` (which clobbered the `0` sentinel with a detected value) was removed, as it is no longer needed.

## Fix 5 — FileWriter always recorded at hardcoded 48000 Hz

**Files:** `src/LiveEffectEngine.cpp`, `src/win32/MainWindow.cpp`

`FileWriter` was constructed with `48000` hardcoded and that value was never updated before `open()`. When the driver negotiated a different sample rate (e.g. 44100 Hz), the recorded file had an incorrect sample-rate header, playing back at the wrong speed.

**Fix:**
- `MainWindow::onEngineStatePoll()` now sets `liveEngine_.sampleRate = audioEngine_.sampleRate()` when the engine transitions to `Running`, keeping the DSP engine's stored rate in sync with the driver-negotiated rate.
- `LiveEffectEngine::startRecording()` sets `fileWriter->sampleRate = sampleRate` immediately before calling `fileWriter->open()`, so the file header reflects the actual runtime rate.

## Fix 6 — Recorded audio high-pitched and metallic

**File:** `src/LiveEffectEngine.cpp`

`LiveEffectEngine::process()` receives stereo-interleaved buffers (`frames * 2` floats), but called `queueManager.process(input, output, frames)` — passing the frame count rather than the sample count. The queue copied only the first half of each audio block. `FileWriter::encode()` then computed `numFrames = pos / channels = frames / 2`, writing half the real content at the declared sample rate. The file played back at 2× speed (one octave up), and the misaligned stereo samples produced metallic distortion.

**Fix:** Changed the call to `queueManager.process(input, output, frames * mOutputChannelCount)` so the full stereo block is copied.

## Fix 7 — Record button text garbled (â---Record)

**File:** `src/win32/ControlBar.cpp`

`setRecordState()` used `SetWindowTextA()` with `\u25cf` and `\u25a0` Unicode escape sequences. The ANSI `SetWindowTextA` API interprets the string as a system code page, so the multi-byte UTF-8 encoding of those code points was displayed as `â---` garbage.

**Fix:** Changed to `SetWindowTextW()` with `L"\u25a0 Stop"` / `L"\u25cf Record"` wide string literals, which pass the characters directly as UTF-16.

---

## Fix 8 — Power-off while recording didn't stop the recording

**File:** `src/win32/MainWindow.cpp`

The `IDC_POWER_TOGGLE` off-branch called `audioEngine_.stop()` and disabled the record button, but did not finalise an in-progress recording. The audio engine was torn down while `FileWriter` still held an open file descriptor, leaving the file potentially truncated or corrupt and `recordingFd_` leaked.

**Fix:** Added recording teardown at the top of the power-off branch — `liveEngine_.stopRecording()`, `_close(recordingFd_)`, and `controlBar_.setRecordState(false)` — before stopping the engine.
