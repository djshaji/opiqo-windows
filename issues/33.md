# Issue 33: WASAPI Latency and Performance Problems

Source: `src/win32/AudioEngine.cpp`

---

## 1. `blockSize` not applied to client buffer period

`impl_->blockSize` is stored but never passed to `IAudioClient::Initialize`. In shared mode both `capturePeriod` and `renderPeriod` are hardcoded to `0`, so the OS picks the default engine period (~10 ms on most systems). The `GetDevicePeriod()` call is exclusive-mode only.

**Fix:** In shared mode, query `GetDevicePeriod(&defaultPeriod, &minPeriod)` and pass the minimum period (or the nearest period that accommodates `blockSize`) as `hnsBufferDuration` to `Initialize`.

---

## 2. `IAudioClient3` low-latency path not used

`IAudioClient3::InitializeSharedAudioStream()` combined with `GetSharedModeEnginePeriod()` enables sub-millisecond shared-mode latency on Windows 10+. The current code activates only `IAudioClient`, capping shared-mode latency at the default engine period.

**Fix:** Query `IAudioClient3` via `IID_IAudioClient3` and fall back to `IAudioClient` on older systems.

---

## 3. Full-buffer pre-roll adds unnecessary startup latency

```cpp
impl_->renderService->ReleaseBuffer(renderBufSize, AUDCLNT_BUFFERFLAGS_SILENT);
```

Silencing the **full** buffer rather than a single device period pre-loads one entire buffer (~10 ms) of silence into the render pipeline before audio starts, adding that duration to startup latency.

**Fix:** Pre-roll only one device period of silence instead of the full buffer size.

---

## 4. `GetStreamLatency()` never queried

The actual hardware round-trip latency reported by WASAPI is never read or surfaced. Callers and the UI have no way to display or act on real latency.

**Fix:** Call `IAudioClient::GetStreamLatency()` on both clients after `Initialize` and expose the sum through `AudioEngine`.

---

## 5. `GetBufferSize()` called inside the inner hot-path loop

```cpp
// Inside while(packetFrames > 0):
UINT32 renderBufSize = 0;
impl_->renderClient->GetBufferSize(&renderBufSize);
```

`GetBufferSize()` returns a value that is constant after `Initialize()`. Calling it on every capture packet adds an unnecessary COM dispatch per packet.

**Fix:** Cache the result in a local variable before the outer `while(true)` loop (alongside the already-cached `captureBufSize`).

---

## 6. Redundant buffer copies in the common stereo-float case

The hot-path pipeline is:

```
WASAPI capture buf → inBuf → stereoBuf → [resampledBuf] → dspOutBuf → outBuf → WASAPI render buf
```

When capture is already stereo float and no sample-rate conversion is needed, the `inBuf → stereoBuf` copy and `dspOutBuf → outBuf` copy are redundant.

**Fix:** Add a fast path: when `capChannels == 2 && isFloatFormat(captureFmt) && !captureResampler`, pass the WASAPI capture pointer directly to `stereoBuf` (or the DSP input), and write directly from `dspOutBuf` to the render buffer when `renChannels == 2 && isFloatFormat(renderFmt)`.

---

## 7. Loop-invariant conditionals inside downmix and upmix tight loops

```cpp
// Downmix — capChannels is invariant over the f loop:
impl_->stereoBuf[f * 2 + 0] = (capChannels >= 1) ? impl_->inBuf[...] : 0.f;
impl_->stereoBuf[f * 2 + 1] = (capChannels >= 2) ? impl_->inBuf[...] : ...;

// Upmix — (c < 2) is invariant for each fixed c:
impl_->outBuf[f * renChannels + c] = (c < 2) ? impl_->dspOutBuf[...] : 0.f;
```

These conditional checks are loop-invariant but evaluated every iteration. The compiler cannot reliably hoist them because the values are read from non-`const` struct fields.

**Fix:** Copy `capChannels` and `renChannels` into `const` locals before the loops and let the compiler (or manual restructure) collapse them.

---

## 8. `renderEvent` registered but never waited on

```cpp
impl_->renderClient->SetEventHandle(impl_->renderEvent);  // event-driven mode enabled

// In runLoop():
HANDLE events[2] = { impl_->captureEvent, impl_->stopEvent };  // renderEvent absent
```

The render client is placed in event-driven mode but its event is never consumed. The render event fires silently, provides no back-pressure, and wastes a kernel event object.

**Fix:** Either remove `SetEventHandle` for the render client (leave it in polling mode), or add `renderEvent` to the `WaitForMultipleObjects` array and handle render-driven writes separately from capture-driven writes.

---

## 9. DSP-processed frames silently discarded when render buffer is full

```cpp
if (available == 0) {
    impl_->dropouts.fetch_add(1, std::memory_order_relaxed);
    continue;  // Processed audio thrown away
}
```

Audio that has already passed through the DSP chain is discarded rather than deferred. This causes audible glitches under any timing jitter, including normal Windows scheduler variance.

**Fix:** Introduce a small lock-free ring buffer between the DSP output and the render write (similar to the existing SRC `resampledBuf` path) so surplus frames are held for the next render period rather than dropped.

---

## 10. No capture–render clock drift correction

The loop wakes on `captureEvent` only and writes synchronously to the render client. Over time, non-zero clock drift between independent capture and render hardware clocks will cause the render buffer to gradually over- or under-fill, resulting in increasing latency or periodic dropouts in long-running sessions.

**Fix:** Periodically measure the render buffer fill level (`GetCurrentPadding`) and use it as a drift signal to slightly adjust the effective write cadence, or use `IAudioClock::GetPosition()` on both streams to track and correct drift.

---

## 11. `resampledBuf` ceiling not derived from resampler filter delay

```cpp
UINT32 maxOut = captureBufSize
    * impl_->renderFmt->nSamplesPerSec
    / impl_->captureFmt->nSamplesPerSec + 128;
```

Integer division truncates before the margin is added, and the 128-frame constant is not based on the actual Speex resampler filter delay (which depends on the chosen quality and rate ratio). This can cause a buffer overflow for high upsample ratios with long filter kernels.

**Fix:** After `speex_resampler_init`, call `speex_resampler_get_output_latency()` and add its value (plus a one-period safety margin) to the computed ceiling instead of the arbitrary 128.

## Build System: MinGW-w64 on Linux

Verified with MinGW-w64 GCC 16.0.1 (Fedora). All required headers are present in
`/usr/x86_64-w64-mingw32/sys-root/mingw/include/audioclient.h`:

- `IAudioClient3`, `IID_IAudioClient3` — defined
- `GetSharedModeEnginePeriod`, `GetCurrentSharedModeEnginePeriod`, `InitializeSharedAudioStream` — defined
- `IAudioClock` — defined
- `IAudioClient::GetStreamLatency` — defined

**All 11 issues can be mitigated** without any additional SDK or stub libraries.
